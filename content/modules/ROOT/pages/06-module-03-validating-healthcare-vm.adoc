With the DR control plane established, it's time to work with the workload that needs protecting. The OpenMRS Foundation's patient management system runs as a virtual machine on OpenShift Virtualization. In this module, you'll confirm the VM is healthy, expose it via an OpenShift Route, and create a sample patient appointment record that you'll use to validate data consistency after the DR failover in Module 5.

IMPORTANT: All exercises in this module must be performed on the **primary cluster (`{primary_cluster}`)** only.

NOTE: All YAML files required for this module are pre-loaded in the lab working directory. You do not need to type them -- simply execute the `oc apply` commands as instructed.

NOTE: The expected outputs shown throughout this lab are examples captured from a reference environment. Your output may display different pod names, IP addresses, timestamps, or resource identifiers. What matters is the context and status.

== Learning objectives

By the end of this module, you'll be able to:

* Verify that a VirtualMachine is running on OpenShift Virtualization using the `oc` CLI
* Expose a VM workload using a Kubernetes Service and an OpenShift Route
* Create application data in the OpenMRS patient management system to serve as a DR validation reference

== Exercise A: Verify the VM is running

Before exposing the application, confirm that the OpenMRS VirtualMachine (`openmrs01`) is running on `{primary_cluster}`.

. Check the VirtualMachine status in the `openmrs` namespace on cluster-PROD:
+
.Top Terminal
[source,console,role="execute-top"]
----
oc get vms -n openmrs
----
+
Expected output:
+
----
NAME       AGE   STATUS    READY
openmrs01  1h    Running   True
----

=== Verify

Confirm the following before continuing:

* `openmrs01` shows `Running` status and `Ready: True` in the `openmrs` namespace

== Exercise B: Generate sample application data

To validate data consistency after the DR failover in Module 5, you need a known record written to persistent storage. You'll expose the application via an OpenShift Route, then create a patient appointment record that you can verify post-failover.

. Verify the Service was created:
+
.Top Terminal
[source,console,role="execute-top"]
----
oc get svc svcopenmrs -n openmrs
----
+
Expected output:
+
----
NAME         TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
svcopenmrs   ClusterIP   10.96.x.x       <none>        80/TCP    30s
----

. Retrieve the Route URL and **note it down** -- you will need this to access the application and again after the DR failover in Module 5:
+
.Top Terminal
[source,console,role="execute-top"]
----
oc get route openmrs -n openmrs
----
+
Expected output:
+
----
NAME      HOST/PORT                                   PATH   SERVICES     PORT   TERMINATION   WILDCARD
openmrs   openmrs-openmrs.<cluster-ingress-domain>          svcopenmrs   80     edge          None
----
+
. Execute the following to output a clickable link to the OpenMRS application:
+
.Top Terminal
[source,console,role="execute-top"]
----
oc get route openmrs -n openmrs --no-headers | awk '{print "https://" $2}'
----

== Learning outcomes

By completing this module, you should now understand:

* How to verify the health of a VirtualMachine resource on OpenShift Virtualization using `oc get vms`
* How a Kubernetes Service with a `kubevirt.io/domain` selector routes traffic to a VM workload
* How an OpenShift Route with TLS edge termination exposes a VM application externally
* Why creating a known data record before DR testing provides a concrete consistency validation reference
