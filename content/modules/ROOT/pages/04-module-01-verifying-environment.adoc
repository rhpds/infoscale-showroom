The OpenMRS Foundation's DR solution can only be as reliable as the infrastructure it runs on. Before configuring any disaster recovery workflows, you need to confirm that InfoScale for Kubernetes is healthy on both clusters and that the required storage configuration is in place.

In this module, you'll verify the InfoScale installation on both the primary cluster (`{primary_cluster}`) and the secondary DR cluster (`{secondary_cluster}`), and confirm that the required StorageClass is available for DR workloads.

NOTE: The expected outputs shown throughout this lab are examples captured from a reference environment. Your output may display different pod names, IP addresses, or timestamps. What matters is the context and status -- for example, that pods are in `Running` state and resources show the expected status.

== Learning objectives

By the end of this module, you'll be able to:

* Confirm that InfoScale for Kubernetes pods are running and healthy on both clusters
* Interpret the `InfoScaleCluster` custom resource status to validate cluster readiness
* Verify that the required InfoScale StorageClass is present and available for DR workloads

== Exercise Z: Set up your user access to the two clusters

There are two clusters in the environment, PROD and DR.
You will use the top terminal for work on *cluster-PROD* and the bonus terminal for work on *cluster-DR*.

. Verify top-terminal access to cluster-PROD
+
[source,role="execute"]
----
oc whoami --show-server
----
+
Expected output:
+
[source,sh,subs="attributes"]
----
https://api.cluster-{guid}.dynamic.redhatworkshops.io:6443
----

. Set up bottom-terminal access to cluster-DR

Change to user `cloud-user` on cluster-DR and setup KUBECONFIG for access to *cluster-DR*:
+
[source,role="execute"]
----
sudo -i
su - cloud-user
export KUBECONFIG=/home/cloud-user/cluster-$GUID-dr/auth/kubeconfig
echo "export KUBECONFIG=/home/cloud-user/cluster-$GUID-dr/auth/kubeconfig" >> ~/.bash_profile
tail -1 ~/.bash_profile
oc whoami --show-server
----
+
Expected output:
+
[source,sh,subs="attributes"]
----
export KUBECONFIG=/home/cloud-user/cluster-{guid}-dr/auth/kubeconfig
https://api.cluster-{guid}-dr.dynamic.redhatworkshops.io:6443
----
+
NOTE: The URI of cluster-DR differs from cluster-PROD only in the hostname, with the `-dr` suffix.

== Exercise A: Verify InfoScale installation on the primary cluster

The OpenMRS Foundation's primary production workloads run on `{primary_cluster}`. Before configuring DR, confirm that InfoScale for Kubernetes {infoscale_version} is fully operational here.

. Verify that the InfoScale pods are running:
+
[source,role="execute"]
----
oc get pods -n infoscale-vtas
----
+
Expected output:
+
----
NAME                                      READY   STATUS    RESTARTS   AGE
infoscale-operator-<id>                   1/1     Running   0          2d
infoscale-csi-controller-<id>             2/2     Running   0          2d
infoscale-csi-node-<id>                   3/3     Running   0          2d
----
+

. Check the InfoScale Cluster status:
+
[source,role="execute"]
----
oc get infoscalecluster -n infoscale-vtas
----
+
Expected output:
+
----
NAME              STATUS   AGE
infoscalecluster  Ready    2d
----

== Exercise C: Verify storage configuration

The `{infoscale_storageclass}` StorageClass is required to provision volumes for the OpenMRS healthcare VM workloads. It must be present and available on **both clusters** before you configure data replication in Module 4.


=== Primary cluster ({primary_cluster})

. List the available StorageClasses:
+
[source,role="execute"]
----
oc get storageclass
----
+
Confirm that `{infoscale_storageclass}` appears in the output. The provisioner column should reference an InfoScale provisioner.

=== Secondary cluster ({secondary_cluster})

. In the terminal logged into `{secondary_cluster}`, list the available StorageClasses:
+
[source,role="execute"]
----
oc get storageclass
----
+
Confirm that `{infoscale_storageclass}` is also present on the secondary cluster.

== Learning outcomes

By completing this module, you should now understand:

* How to confirm that InfoScale for Kubernetes is operational using the `InfoScaleCluster` CRD
* What a healthy InfoScale pod deployment looks like in the `{infoscale_namespace}` namespace
* Why verifying storage configuration on both clusters is a prerequisite for DR setup
* The baseline environment state required before proceeding with DR component installation
