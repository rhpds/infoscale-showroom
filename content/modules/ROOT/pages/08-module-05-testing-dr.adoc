This is the moment the OpenMRS Foundation has been preparing for. In this module, you'll validate the end-to-end disaster recovery execution on `{secondary_cluster}`. You'll confirm that the healthcare VM migrated successfully, re-expose the application service, and verify that the patient appointment record created in Module 3 is fully intact -- proving that InfoScale's asynchronous replication delivered near-zero data loss through continuous non-periodic replication.

IMPORTANT: All exercises in this module must be performed on the **secondary cluster (`{secondary_cluster}`)** only.

NOTE: All YAML files required for this module are pre-loaded in the lab working directory. You do not need to type them -- simply execute the commands as instructed.

NOTE: The expected outputs shown throughout this lab are examples captured from a reference environment. Your output may display different pod names, IP addresses, or timestamps. What matters is the context and status.

== Learning objectives

By the end of this module, you'll be able to:

* Confirm that a VirtualMachine workload is running on the DR cluster after a failover
* Re-expose a VM application service and route on the secondary cluster
* Validate application data consistency by verifying records created before the failover are intact
* Interpret a successful end-to-end DR test as confirmation of near-zero data loss through continuous non-periodic replication

== Exercise A: Verify the VM is running on the secondary cluster

After the migration triggered in Module 4, the `openmrs01` VirtualMachine should be running on `{secondary_cluster}`. Confirm this before attempting to expose or access the application.

. Verify that the VirtualMachine is running on cluster-DR:
+

----

openmrs01  1h    Running   True
----

== Exercise B: Expose the VM service on the secondary cluster

The Service and Route created in Module 3 existed only on `{primary_cluster}`. You need to recreate them on `{secondary_cluster}` so the application is accessible at the DR site.

. Apply the same service and route manifest used in Module 3 on cluster-DR:
+
[source,yaml,role="execute-bottom"]
----
oc apply -f - <<EOF
apiVersion: v1
kind: Service
metadata:
  name: svcopenmrs
  namespace: openmrs
spec:
  ports:
    - port: 80
      targetPort: 80
      protocol: TCP
  selector:
    kubevirt.io/domain: openmrs01
---
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: openmrs
  namespace: openmrs
spec:
  to:
    kind: Service
    name: svcopenmrs
  port:
    targetPort: 80
  tls:
    termination: edge
EOF
----

. Verify the Service was created:
+
[source,console,role="execute-bottom"]
----
oc get svc svcopenmrs -n openmrs
----
+
Expected output:
+
----
NAME         TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
svcopenmrs   ClusterIP   10.96.x.x       <none>        80/TCP    30s
----

. Retrieve the Route URL:
+
[source,console,role="execute-bottom"]
----
oc get route openmrs -n openmrs
----
+
Expected output:
+
----
NAME      HOST/PORT                                    PATH   SERVICES     PORT   TERMINATION   WILDCARD
openmrs   openmrs-openmrs.<cluster-dr-ingress-domain>         svcopenmrs   80     edge          None
----
+
. Execute the following to output a clickable link to the OpenMRS application:
+
.Top Terminal
[source,console,role="execute-bottom"]
----
oc get route openmrs -n openmrs --no-headers | awk '{print "https://" $2}'
----

=== Verify

Confirm the following before continuing:

* `svcopenmrs` Service and `openmrs` Route are present in the `openmrs` namespace on `{secondary_cluster}`
* You have noted the new Route URL for `{secondary_cluster}`

== Learning outcomes

By completing this module, you should now understand:

* How to confirm a successful VM workload migration by checking VirtualMachine status on the DR cluster
* Why Service and Route resources must be recreated on the secondary cluster after a failover (they are cluster-scoped, not replicated by InfoScale)
* How application-level data validation provides stronger DR confirmation than infrastructure checks alone
* What "near-zero data loss through continuous non-periodic replication" means in practice for an asynchronous replication scenario -- and how to demonstrate it with a concrete validation test
